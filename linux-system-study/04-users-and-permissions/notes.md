# 用户和权限基础笔记

这一章是 Linux 学习里一个非常关键的转折点。

前面几章你已经知道怎么创建文件、查看文件、搜索文本。到这里要开始回答一个更现实的问题：

- 谁能看这个文件
- 谁能改这个文件
- 谁能进入这个目录
- 权限不够时到底卡在什么地方

如果这一章理解得比较扎实，后面学服务、日志、安全、部署都会顺很多。

## 1. 为什么 Linux 一定要有权限模型

Linux 经常是多用户系统。

这意味着同一台机器上可能同时有：

- 多个普通用户
- 一些系统服务账户
- 管理员账户

如果没有权限模型，就会出现这些问题：

- 谁都能改任何配置
- 谁都能删别人的文件
- 谁都能读取不该读的数据

所以 Linux 需要一套规则，去决定“谁能对什么做什么”。

## 2. 用户是什么

用户可以先简单理解成：系统里的一个身份。

你在终端里执行命令时，系统不是只看“执行了什么”，还会看“是谁执行的”。

最常见的两类用户：

- 普通用户
- `root`

### 普通用户

普通用户适合做日常工作，例如：

- 编辑自己的文件
- 运行自己的程序
- 查看自己有权限看的内容

### root

`root` 是超级用户，拥有极高权限。

它通常可以：

- 修改系统配置
- 安装软件
- 改其他用户文件的所有者
- 管理系统服务

一个重要原则是：

- 平时尽量用普通用户做事
- 只有确实需要系统级改动时才使用更高权限

## 3. 用户组是什么

用户组可以理解成一组用户的集合。

它的意义在于：很多权限不需要一条条给单个用户配置，而是可以按组来管理。

例如：

- 开发组成员可以共同访问某个项目目录
- Web 服务组可以共同访问某些站点文件

你可以先记住：

- 用户是个人身份
- 用户组是方便批量管理权限的一层

## 4. 用户、组和文件之间的关系

每个文件或目录通常至少和这些信息有关：

- 所有者
- 所属组
- 权限位

例如：

```text
-rw-r--r-- 1 zcy staff 120 Apr 21 10:00 notes.txt
```

可以先拆成几块来看：

- `-`：文件类型
- `rw-r--r--`：权限位
- `zcy`：所有者
- `staff`：所属组

这意味着系统在做权限判断时，通常会先问：

1. 你是不是所有者
2. 如果不是，你是不是属于这个组
3. 如果还不是，就按“其他用户”的权限处理

## 5. 如何看懂 `rwx`

权限位通常分成三组：

```text
rw- r-- r--
```

它们分别表示：

- 第一组：所有者权限
- 第二组：所属组权限
- 第三组：其他用户权限

每组三个字符分别是：

- `r`：read，读
- `w`：write，写
- `x`：execute，执行

如果某一位没有权限，就会显示成 `-`。

例如：

- `rw-`：可读可写，不可执行
- `r--`：只读
- `rwx`：可读可写可执行

## 6. 文件权限和目录权限的含义不完全一样

这是初学者最容易混淆的地方之一。

### 对普通文件来说

- `r`：可以读取文件内容
- `w`：可以修改文件内容
- `x`：可以把它当作程序或脚本执行

### 对目录来说

- `r`：可以查看目录里的名字列表
- `w`：可以在目录里增删改条目
- `x`：可以进入这个目录，或通过它访问下面的对象

这里最关键的是目录的 `x`。

很多人第一次觉得奇怪，是因为：

- 目录有 `r` 但没有 `x`，可能看得到名字却进不去
- 文件本身权限没问题，但上层目录没有合适权限，也一样访问失败

所以排查权限时，不能只看文件本身，还要看目录链路。

## 7. 数字权限怎么理解

Linux 里常见数字权限写法：

```bash
chmod 644 notes.txt
chmod 755 run.sh
```

数字的来源是：

- `r = 4`
- `w = 2`
- `x = 1`

所以：

- `7 = 4 + 2 + 1 = rwx`
- `6 = 4 + 2 = rw-`
- `5 = 4 + 1 = r-x`
- `4 = r--`

### `644`

表示：

- 所有者：读写
- 组用户：只读
- 其他用户：只读

常见于普通文本文件。

### `755`

表示：

- 所有者：读写执行
- 组用户：读执行
- 其他用户：读执行

常见于目录，或者需要执行的脚本。

一个很实用的直觉是：

- 普通文件常见 `644`
- 目录常见 `755`

但这只是常见值，不是硬性规则。

## 8. 符号权限写法也很重要

除了数字写法，还可以用符号方式改权限。

例如：

```bash
chmod u+x run.sh
chmod g-w shared.txt
chmod o-r secret.txt
```

这里：

- `u`：user，所有者
- `g`：group，组用户
- `o`：others，其他用户
- `a`：all，所有人

符号的好处是更直观，尤其适合小范围调整。

例如：

- `u+x`：给所有者加执行权限
- `g-w`：去掉组写权限
- `a+r`：给所有人加读权限

## 9. `chmod`、`chown`、`chgrp` 分别做什么

这三个命令名字很像，但解决的问题不同。

### `chmod`

改的是权限位。

例如：

```bash
chmod 644 notes.txt
chmod u+x run.sh
```

### `chown`

改的是所有者，必要时也可以一起改所属组。

例如：

```bash
sudo chown zcy notes.txt
sudo chown zcy:staff notes.txt
```

### `chgrp`

只改所属组。

例如：

```bash
chgrp staff notes.txt
```

可以记成：

- `chmod`：改“能做什么”
- `chown`：改“属于谁”
- `chgrp`：改“属于哪个组”

## 10. `sudo` 到底是什么

`sudo` 不是“更高级的命令模式”，而是“以更高权限执行这一条命令”。

例如：

```bash
sudo apt update
sudo chown root:root file.txt
```

它的核心意义是：

- 不是永久切到 root
- 而是临时借更高权限执行一次操作

一个很重要的使用原则：

- 能不用 `sudo` 就别用
- 只有确定是系统级操作时再用

为什么不能滥用：

- 容易误改系统文件
- 容易把本来属于普通用户的文件改成 root 所有
- 出问题时更难排查

## 11. `umask` 是什么

`umask` 可以先理解成：创建新文件和新目录时，默认权限会被扣掉哪些位。

例如你新建一个文件，并不是一开始就默认 `777` 或 `666` 直接可用，而是会受到 `umask` 影响。

查看当前值：

```bash
umask
umask -S
```

初学阶段你先知道两件事就够：

- 它会影响新创建文件和目录的默认权限
- 它解释了为什么你新建的文件常常不是“所有权限都打开”

## 12. 常见排查思路：为什么会 `Permission denied`

看到权限报错时，不要只盯着命令本身，先按顺序想：

1. 我现在是谁
2. 这个文件或目录属于谁
3. 我的用户是否属于它的组
4. 权限位有没有允许这次操作
5. 上层目录有没有给我通过权限
6. 这件事是否本来就需要 `sudo`

很常见的几类情况：

- 文件可读，但目录不可进入
- 文件属于 root，普通用户不能写
- 脚本没有执行权限
- 目录没有写权限，所以即使文件名存在也无法创建新文件

## 13. 目录权限为什么经常比文件权限更难理解

因为对目录来说，权限控制的是“操作目录条目”的能力，而不是“读目录内容文本”。

例如一个目录如果没有写权限，你往往不能：

- 在里面新建文件
- 删除里面文件
- 重命名里面文件

即使目录里的某个文件本身是你能读的，目录权限也仍然会影响整体访问。

所以很多权限问题不是单点问题，而是“路径链路”问题。

## 14. 这一章里最值得建立的几个直觉

- 权限判断永远和“当前用户是谁”有关
- 文件和目录的 `rwx` 含义不一样
- `chmod` 是改权限位，不是改所有者
- `sudo` 是临时提权，不是默认工作方式
- 看到 `Permission denied` 时，先查用户、所有者、权限位，再考虑是否提权

## 15. 这一章学完后应该形成什么感觉

如果这一章学得顺，你会慢慢形成这些感觉：

- `ls -l` 不再是一串难记字符，而是一段可读信息
- 看到 `644`、`755` 会开始有直觉
- 目录权限不再只会死记，而能理解为什么会影响访问
- 遇到权限报错时，不会第一反应就盲目 `sudo`

到这里，你就开始真正接近 Linux 系统管理的核心能力了。
