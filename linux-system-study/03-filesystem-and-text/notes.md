# 文件系统和文本处理笔记

这一章有两个核心主题：

- 文件系统是什么，Linux 里的文件和目录到底怎么组织
- 文本为什么是 Linux 里最重要的信息载体，以及怎么快速处理它

这两件事放在一起学非常自然，因为 Linux 里大量问题最后都会落到：

- 去哪里找文件
- 怎么看文件内容
- 怎么从文件里筛出需要的信息

## 1. 文件系统是什么

可以先把文件系统理解成：操作系统组织和管理文件的一套规则。

它回答的是这些问题：

- 文件放在哪里
- 目录之间怎么层层组织
- 文件有哪些属性
- 系统怎么找到你要的那个文件

你平时看到的这些路径：

```bash
/home/zcy
/etc/hosts
/var/log
```

背后都属于文件系统的一部分。

## 2. 目录并不是“文件夹皮肤”

在 Linux 里，目录不只是图形界面里看到的“文件夹”，它本质上也是文件系统中的一种对象。

你可以先形成一个实用理解：

- 普通文件：主要用来存内容
- 目录：主要用来组织名字和路径
- 链接：主要用来指向别的文件

这也是为什么很多命令既能处理文件，也能处理目录，只是行为略有不同。

## 3. 路径是文件系统的导航方式

你之前已经接触过路径，这里再往前一步理解：

- 路径不是可有可无的字符串
- 它是系统定位对象的方式

例如：

```bash
/var/log/syslog
```

表示从根目录 `/` 开始，一层层进入：

1. `var`
2. `log`
3. `syslog`

Linux 里很多问题都可以先问自己：

- 我是在找路径，还是在找内容
- 我知道文件在哪吗
- 我知道文件里要找什么吗

## 4. 常见文件类型先建立印象

用 `ls -l` 时，最左边第一个字符经常会暴露文件类型。

例如：

```text
-rw-r--r-- 1 zcy staff 120 Apr 21 10:00 notes.txt
drwxr-xr-x 3 zcy staff  96 Apr 21 10:01 docs
lrwxr-xr-x 1 zcy staff  10 Apr 21 10:02 latest -> notes.txt
```

你可以先这样记：

- `-`：普通文件
- `d`：目录
- `l`：符号链接

初学阶段先认识这三种就够用很多场景了。

## 5. 文件不只有内容，还有元数据

一个文件除了“里面写了什么”，还会有很多附加信息，例如：

- 文件名
- 大小
- 修改时间
- 所有者
- 权限
- 文件类型

这些信息通常叫元数据。

所以“看文件”有时不是看内容，而是看属性。

例如你可能会问：

- 这个文件多大
- 它最近什么时候被改过
- 它是文本还是二进制

这些问题对应的就不是 `cat`，而更可能是 `ls -l`、`file`、`stat`

## 6. 为什么 Linux 里文本这么重要

Linux 里很多信息都以文本形式存在：

- 配置文件是文本
- 日志通常是文本
- 命令输出大多也是文本

这意味着，学 Linux 不只是“会打开文件”，更重要的是会：

- 快速浏览
- 只看头尾
- 搜关键字
- 统计和整理结果

这就是为什么 `cat`、`less`、`grep`、`wc` 这些命令会高频出现。

## 7. 查看文件内容时，不是只有一种看法

### `cat`

适合短文件，直接把全部内容输出出来。

### `less`

适合长文件，分页看，而且支持搜索。

### `head`

只看开头几行，适合快速确认文件结构。

### `tail`

只看最后几行，特别适合日志文件。

一个很实用的经验是：

- 不确定文件长短时，先用 `head`
- 文件很长时，优先 `less`
- 看日志最新内容时，优先 `tail`

## 8. 文本搜索和文件搜索不是一回事

这是初学者很容易混淆的点。

### 查文件：`find`

你知道自己要找的是“文件本身”，例如：

- 名字叫 `app.log`
- 后缀是 `.conf`
- 只想找目录

这时常用的是：

```bash
find . -name "*.log"
find . -type d -name "scripts"
```

### 查内容：`grep`

你知道自己要找的是“文本里的一段内容”，例如：

- 哪一行里出现了 `ERROR`
- 哪些文件包含 `TODO`

这时常用的是：

```bash
grep "ERROR" app.log
grep -r "TODO" .
```

可以记一个很实用的区分：

- `find` 是找对象
- `grep` 是找内容

## 9. 日志排查为什么经常用 `tail` + `grep`

日志文件通常有两个特点：

- 很长
- 最新信息通常在末尾

所以常见思路是：

1. 先用 `tail` 看最近几行
2. 再用 `grep` 搜关键字

例如：

```bash
tail -n 20 app.log
grep -in "error" app.log
```

如果日志还在持续增长，还会用：

```bash
tail -f app.log
```

这会持续跟踪文件末尾的新内容。

## 10. 文本统计和整理的基本动作

文本处理不一定都是复杂脚本，很多时候只是做几步基础整理。

### `wc`

回答“有多少”：

- 多少行
- 多少单词
- 多少字节

### `sort`

回答“能不能排好序”。

### `uniq`

回答“有没有重复行”。

一个典型例子：

```bash
sort names.txt | uniq
```

含义是：

- 先排序
- 再把相邻的重复行合并

为什么常常要先 `sort` 再 `uniq`？

因为 `uniq` 只会对相邻的重复行起作用，不会跨很远去找重复项。

## 11. 文本处理的常见思路

遇到一个文件时，你可以按这个顺序想：

1. 这是个什么文件
2. 文件在哪
3. 文件有多大
4. 我要看全部，还是只看部分
5. 我要找名字，还是找内容
6. 我要看单次结果，还是持续跟踪

这会比一上来就乱试命令更稳定。

## 12. 文件系统学习里值得先知道的两个概念

### 隐藏文件

在 Linux 里，以 `.` 开头的文件通常会被当作隐藏文件。

例如：

- `.bashrc`
- `.gitignore`

它们不是特殊格式，只是默认不会被普通 `ls` 显示。

### 链接

最常先建立印象的是符号链接，也就是软链接。

例如：

```bash
ln -s notes.txt latest.txt
```

可以先把它理解成“一个指向别处的快捷入口”。

初学阶段先知道它存在，以及 `ls -l` 能看到箭头关系，就够了。

## 13. 初学者常见坑

### 用 `cat` 打开超大文件

这样可能一下刷满屏幕，信息反而难看。

### 把 `find` 和 `grep` 混掉

找文件名用 `find`
找文件内容用 `grep`

### 忘了 `tail -f` 会一直挂着

它不是“看一下就结束”，而是会持续监听，通常要按 `Ctrl + C` 退出。

### 以为 `uniq` 会自动全局去重

其实它默认只处理相邻重复项，所以经常要配合 `sort`。

## 14. 这一章学完后应该形成什么感觉

如果这一章学得顺，你会开始形成这些直觉：

- 文件系统不只是目录树，而是一套组织对象的规则
- 看文件内容时，应该根据场景挑工具，而不是只会 `cat`
- 查文件和查内容是两条不同思路
- 文本处理常常是“查看 + 搜索 + 统计 + 整理”的组合

有了这些直觉，后面学习权限、日志、服务、网络排查都会顺很多。
